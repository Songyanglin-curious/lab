变长长度编码可以理解成“128 进制”的长度表示。每个长度字节有 8 位，但最高位不用来表示数值，而是作为“后面是否还有长度字节”的标志；只有低 7 位参与计算。所以每多一个长度字节，就多 7 位表示能力。比如 300 可拆成 `[2,44]`，即 `2×128+44`，传输时变成 `[0x82,0x2C]`：`0x82` 表示数值 2 且后面还有，`0x2C` 表示数值 44 且长度结束。


一个字节代表128 比如300 就是 第一个字节2 代表2\*128  第二个字节代表个位就是300%128的余数

以首位比特判断变长后续还有无表示长度的字节 1 代表有 0代表没有


传输帧只要字节流 所有的业务层主动向传输帧接口靠拢，为了能够让各个协议都支持 所以使用了IAsdu 接口来规范和限制这一约束必须实现。

`NetworkStream.Read(...)` 每读走几个**字节**，这几个字节就从 TCP 接收缓冲里被“消费掉”了。下一次 `Read` 会从后面的数据继续读。

TCP 保证字节流可靠、有序、不丢、不乱序；但不保证一次 `Read` 就把你想要的字节数全部给你。


`FrameCodec + Exchange` 抽象的是“面向业务消息的可靠帧收发能力”，它把底层 TCP 字节流包装成一条条可识别的业务消息。


IAsdu
= 我要发什么业务内容

FrameCodec
= 业务内容怎么包装成一帧、怎么从一帧里拆出来

Exchange
= 这一帧字节通过什么通道发出去、怎么读回来

Exchange + FrameCodec = MessageTransport，消息传输能力

业务代码
↓
MessageTransport：发送一条业务消息 / 接收一条业务消息
↓
FrameCodec：把消息变成帧，或从帧里拆消息
↓
Exchange：把帧字节真正发出去 / 读回来
↓
TCP / TLCP / Mock


业务代码调用 `FrameCodec` 发一帧；`FrameCodec` 不亲自联网，只调用 `Exchange` 把这一帧字节发出去。


`Net` 既管理连接创建方式，也在 `FrameCodec` 之上提供了一层“发送业务消息并接收响应”的封装。调用方只关心发什么、发到哪、收到什么，不关心帧协议和底层通道。
