# 网络基础 02 — TCP 协议

> 目标：理解 TCP 的核心设计——**可靠、面向连接、字节流**。尤其是"字节流"这个特性，它是整套通信框架设计的起点。

---

## 1. 问题引入

你写好了 Client，往 Server 发了 "GetChannelStatus"，Server 收到了，回了个 "通道状态：已开启"——一切正常。

但如果 Client **连发两条消息**："GetChannelStatus" 和 "GetFileList"，Server 怎么区分？收到的可能是 `"GetChannelStatus"+"GetFileList"` 拼在一起的一个大字符串。

这就是 TCP 的**字节流**特性导致的问题。要理解这个问题，先要理解 TCP 是什么。

---

## 2. TCP 是什么

TCP（Transmission Control Protocol，传输控制协议）在网络世界里的定位：

```
┌─────────────────────────┐
│  应用层  你的代码/HTTP/... │  ← 你写代码的层面
├─────────────────────────┤
│  传输层  TCP / UDP      │  ← 我们讨论的层面
├─────────────────────────┤
│  网络层  IP 地址/路由     │
├─────────────────────────┤
│  链路层  网卡/MAC 地址    │
└─────────────────────────┘
```

TCP 要解决的核心问题：两个程序之间**可靠**地传输数据。

### 2.1 TCP 的三个保证

| 保证 | 含义 | 怎么做到的 |
|------|------|-----------|
| 有序 | 发送顺序 = 接收顺序 | 每个字节都有序号 |
| 可靠 | 保证到达，丢包会重传 | 接收方回 ACK 确认 |
| 面向连接 | 先建立连接再传数据 | 三次握手 |

### 2.2 三次握手（建立连接）

```
Client                           Server
  |                                |
  |──SYN (我要连你)──────────────>|
  |                                |
  |<─SYN+ACK (收到了，你也连我)───|
  |                                |
  |──ACK (好，连上了)─────────────>|
  |                                |
  |======= 开始传数据 ============|
```

三次握手由操作系统完成，你的代码只需要调用 `new TcpClient("ip", port)`，OS 在背后做握手。

### 2.3 四次挥手（断开连接）

```
Client                           Server
  |──FIN (我要关了)──────────────>|
  |<─ACK (知道了)─────────────────|
  |                                |
  |<─FIN (我也关了)───────────────|
  |──ACK (好)────────────────────>|
```

这个也不用手动写。`client.Close()` 触发 OS 完成挥手。

## 3. 核心概念：TCP 是字节流，无消息边界

这是理解 **一切通信框架设计** 的钥匙。

### 3.1 什么是"流"？

```
类比：往水管里倒水

  你分三次往水管里倒：
    第 1 次：倒一杯红色液体
    第 2 次：倒一杯蓝色液体
    第 3 次：倒一杯绿色液体

  对面水龙头打开：
    接到什么？——混合液体。分不出哪里是红色哪里是蓝色。

  这就是 TCP 的"流"本质：
    你分三次 Write("AAAA")、Write("BBBB")、Write("CCCC")
    对面可能一次 Read 就收到 "AAAABBBBCCCC"
```

### 3.2 两个典型问题

```
粘包（消息粘在一起）：
  发送端：Write("HELLO")  →  Write("WORLD")
  接收端：Read() → "HELLOWORLD"    ← 两条消息粘成一条

拆包（一条消息被拆成多段）：
  发送端：Write("HELLO_WORLD_HELLO_WORLD")  ← 一条长消息
  接收端：Read() → "HELLO_WOR"              ← 第一次读了前半段
          Read() → "LD_HELLO_WORLD"         ← 第二次读了后半段
```

为什么会粘包/拆包？

1. **Nagle 算法**：OS 为了省带宽，把多个小包攒到一起再发
2. **缓冲区**：发送缓冲区和接收缓冲区都是"仓库"，你写 3 次，OS 可能一次性取走
3. **网络 MTU**：一个包最大 1500 字节左右，消息太长会被 OS 拆成多段

### 3.3 关键结论

```
TCP 只保证：字节按顺序到达，一个不落。
TCP 不保证：你 Write 几次，对方就 Read 几次。
            Write 的"次数"和 Read 的"次数"没有对应关系。
```

## 4. 这就是 RemoteOps 为什么要帧协议

既然 TCP 没有消息边界，怎么让接收方知道"一条消息在哪结束"？

**加帧头**：在每条消息前贴一个标签，写清楚"我多长"。

```
不加帧协议：
  发送：AAAA BBBB CCCC
  接收：AAAABBBBCCCC  ← 谁知道怎么拆？

加帧协议（EB90 + 长度）：
  发送：[EB90][4]AAAA  [EB90][4]BBBB  [EB90][4]CCCC
  接收：先读 2 字节 → 发现是 EB90 → 说明这是新消息的开始
        再读长度 → 4 字节 → 然后读 4 字节数据 = "AAAA"
        再读 2 字节 → 又是 EB90 → 下一条...
```

这就是 `APDU.cs` 存在的根本原因。TCP 没有给你消息边界，你自己加。

---

## 5. 常见坑

| 坑 | 解释 |
|----|------|
| 以为 Write 一次 = Read 一次 | 不是。TCP 流没有次数对应关系 |
| 以为 Read 一定会读到请求的字节数 | 不一定。`Read(100)` 可能只返回 30，需要循环读 |
| localhost 上不容易看到粘包 | 回环接口有优化，粘包不明显。真实的局域网/VPN 延迟大，更容易粘 |
| TCP 和 UDP 的区别 | TCP 像打电话（要接通，能确认对方听到）；UDP 像寄明信片（丢了你也不知道） |
