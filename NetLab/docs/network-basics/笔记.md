`stream.Flush()` 简单说就是**把临时攒在内存（缓冲区）里的数据，强行立刻写到目标位置**（比如文件、网络流、控制台等），并把缓冲区清空。

**总结一句话**：`Flush` 是催促 Stream “别墨迹了，赶紧把手里攒的数据都真正写出去”。大部分情况下靠 `Close`/`Dispose` 可以自动完成，但在需要实时性、可靠性和跨进程通信时必须主动调用。

TCP 本身没有“消息次数”的概念，只有“字节流”

核心变化就是：不要再用“我发了几次”来理解 TCP，要改成“我往一条管道里塞了多少字节

我写入的是字节；
对方读取的是当前缓冲区里可读的一段字节；
至于这一段对应几条业务消息，TCP 不管。

TCP 层只关心字节

TCP 传的是连续字节流；字符串、JSON、命令字段都是应用层解释。每次 `Read()` 得到的只是字节片段，不能直接等同于一个完整字符串或一条完整消息。
