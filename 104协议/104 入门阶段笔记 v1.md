# IEC 104 协议入门笔记 v1

```text

1.Frame层：I/S/U帧
2.控制域：VS/VR序号确认机制
3. ASDU固定头：TypeId/VSQ／COT /CommonAddress
4.信息体： IOA + Value + Quality + Time
5.常用TypeId：遥信、遥测、遥控、总召唤、对时
6，基本流程：启动、总召唤、变化上送、遥控、心跳
```

## 1. 104 的核心理解

IEC 104 可以先理解为：

```text
TCP 长连接 + 104 自己的帧结构 + 电力远动点数据
```

TCP 只负责提供字节流通道，104 自己还要解决：

```text
1. 通信是否启动
2. 数据帧如何确认
3. 点数据如何表达
```

104 不是直接传业务对象，而是传：

```text
站址 + 点号 + 值
```

具体“这个点号代表什么设备、什么状态”，需要靠点表解释。

---

## 2. 104 报文基本结构

104 报文一般长这样：

```text
68 + 长度 + C1 C2 C3 C4 + ASDU可选
```

含义：

| 部分            | 含义                                    |
| --------------- | --------------------------------------- |
| `68`          | 启动字符                                |
| `长度`        | 后续字节数，不包含 `68`和长度字节本身 |
| `C1 C2 C3 C4` | 控制域                                  |
| `ASDU`        | I 帧里的业务数据，S 帧/U 帧没有 ASDU    |

所以 Frame 层负责：

```text
读 68
读长度
读控制域
判断 I / S / U 帧
如果是 I 帧，再继续解析 ASDU
```

---

## 3. I / S / U 三种帧

104 有三类帧：

| 帧   | 作用                 | 是否有 ASDU |
| ---- | -------------------- | ----------- |
| I 帧 | 传业务数据           | 有          |
| S 帧 | 确认收到对方 I 帧    | 无          |
| U 帧 | 启动、停止、测试链路 | 无          |

判断帧类型主要看 `C1`：

```csharp
if ((C1 & 0x01) == 0)
{
    // I 帧
}
else if ((C1 & 0x03) == 0x01)
{
    // S 帧
}
else if ((C1 & 0x03) == 0x03)
{
    // U 帧
}
```

核心记忆：

```text
I/S/U 是帧类型，属于 Frame 层；
ASDU 是 I 帧里的内容，不负责抽象 S 帧和 U 帧。
```

---

## 4. U 帧：启动通信

TCP 连接建立后，104 还不能直接传业务数据，需要先启动 104 数据传输。

典型流程：

```text
主站：STARTDT act
子站：STARTDT con
```

对应报文：

```text
发送：68 04 07 00 00 00
接收：68 04 0B 00 00 00
```

含义：

```text
TCP 已连接
  ↓
主站发 STARTDT act，请求启动数据传输
  ↓
子站回 STARTDT con，确认启动
  ↓
后续才开始传 I 帧业务数据
```

注意：

```text
STARTDT con 确认的是“104 数据传输可以开始”；
它不是确认某一条业务数据。
```

---

## 5. S 帧：确认 I 帧

S 帧是纯确认帧，不传业务数据。

典型格式：

```text
68 04 01 00 XX XX
```

例如：

```text
68 04 01 00 02 00
```

含义：

```text
01 00：说明这是 S 帧
02 00：接收序号
```

104 里的序号不是直接裸写，而是：

```text
实际序号 << 1
```

所以：

```text
02 00 = 0x0002
0x0002 >> 1 = 1
```

表示：

```text
我已经收到你序号 0 的 I 帧；
我下一条期望收到序号 1 的 I 帧。
```

重要区别：

| 报文        | 确认什么               |
| ----------- | ---------------------- |
| STARTDT con | 确认 104 传输启动      |
| S 帧        | 确认收到对方 I 帧      |
| ASDU 确认   | 确认具体业务命令被接受 |

一句话：

```text
STARTDT con 是开工确认；
S 帧是收货确认。
```

---

## 6. I 帧控制域：VS / VR

I 帧控制域：

```text
C1 C2 C3 C4
```

分成两部分：

```text
C1 C2 = 发送序号 VS
C3 C4 = 接收序号 VR
```

计算规则：

```text
VS = 小端读取(C1 C2) >> 1
VR = 小端读取(C3 C4) >> 1
```

例如：

```text
C1 C2 = 00 00 → VS = 0
C1 C2 = 02 00 → VS = 1
C1 C2 = 04 00 → VS = 2
```

I 帧既传业务，也可以顺便确认对方的 I 帧。

所以：

```text
S 帧 = 只确认，不传业务
I 帧 = 传业务，同时也带确认序号
```

---

## 7. ASDU：I 帧里的业务数据

I 帧结构：

```text
68 长度 C1 C2 C3 C4 ASDU...
```

ASDU 可以先理解成：

```text
类型 + 原因 + 站址 + 点号 + 值
```

常见结构：

```text
TypeId + VSQ + COT + CommonAddress + 信息体
```

对应中文：

| 字段          | 含义                                               |
| ------------- | -------------------------------------------------- |
| TypeId        | 类型标识，说明这是什么业务                         |
| VSQ           | 可变结构限定词，说明有几个信息体，以及地址是否连续 |
| COT           | 传送原因，说明为什么发                             |
| CommonAddress | 公共地址，表示哪个站/装置                          |
| 信息体        | IOA + Value                                        |

---

## 8. 总召唤 ASDU 示例

报文：

```text
68 0E 00 00 00 00 64 01 06 00 01 00 00 00 00 14
```

先去掉 104 外壳：

```text
68 0E 00 00 00 00
```

剩下 ASDU：

```text
64 01 06 00 01 00 00 00 00 14
```

拆解：

```text
64        TypeId = 总召唤
01        VSQ = 1 个信息体
06 00     COT = 激活
01 00     CommonAddress = 1 号站
00 00 00  IOA = 0
14        QOI = 总召唤限定词
```

整体含义：

```text
主站向 1 号站发送总召唤命令，请求它上送当前全量数据。
```

---

## 9. VSQ：一个 ASDU 可以带多个点

一帧 ASDU 不一定只带一个点，数量由 VSQ 决定。

VSQ 包含两个信息：

```text
低 7 位：信息体数量
最高位：地址是否连续
```

### 非连续地址

如果 VSQ 最高位是 0：

```text
IOA1 + Value1
IOA2 + Value2
IOA3 + Value3
```

每个点都带自己的 IOA。

### 连续地址

如果 VSQ 最高位是 1：

```text
起始IOA + Value1 + Value2 + Value3
```

只给第一个 IOA，后面的 IOA 自动递增。

---

## 10. IOA：信息体地址

IOA 全称：

```text
Information Object Address
```

可以先理解成：

```text
IOA = 点号
```

它通常是 3 字节，小端序。

例如：

```text
05 00 00 → IOA = 5
01 07 00 → IOA = 0x701
```

完整定位一个点，通常要靠：

```text
CommonAddress + IOA
```

比如：

```text
公共地址 = 1
IOA = 5
Value = 1
```

协议层只能知道：

```text
1号站的5号点，值为1
```

至于这个点代表“某开关合位”还是“某通道在线”，要查点表。

---

## 11. 点表的作用

104 报文负责传：

```text
站址 + 点号 + 值
```

点表负责解释：

```text
这个点号代表什么业务含义
```

例如：

| 公共地址 | IOA | 类型 | 名称            | 值含义     |
| -------- | --: | ---- | --------------- | ---------- |
| 1        |   5 | 遥信 | 1号线路开关位置 | 0=分，1=合 |

所以：

```text
104 报文不是点表；
104 报文是运行时数据；
点表是点号到业务含义的映射表。
```

## 13. 常用 TypeId：遥信

遥信就是状态量，例如：

```text
开 / 关
分 / 合
在线 / 离线
正常 / 告警
投入 / 退出
```

常见遥信 TypeId：

| TypeId | 名称     | 含义              |
| -----: | -------- | ----------------- |
| `01` | 单点遥信 | 一个 bit 表示状态 |
| `03` | 双点遥信 | 两个 bit 表示状态 |

### 13.1 单点遥信 TypeId = 01

单点遥信信息体结构：

```text
IOA 3字节 + SIQ 1字节
```

其中：

```text
IOA = 点号
SIQ = 状态值 + 品质位
```

先不管品质位，只看最低位：

```text
SIQ & 0x01
```

含义：

|    值 | 含义                 |
| ----: | -------------------- |
| `0` | 分 / 关 / 否 / false |
| `1` | 合 / 开 / 是 / true  |

例如：

```text
03 00 00 01
```

拆解：

```text
03 00 00    IOA = 3
01          SIQ = 1
```

含义：

```text
3号遥信点 = 1
```

具体这个点表示什么，需要查点表。

### 13.2 单点遥信帧示例

```text
68 1A 02 00 02 00 01 04 14 00 01 00
03 00 00 00
05 00 00 00
08 00 00 01
09 00 00 00
```

拆 Frame：

```text
68              启动符
1A              后续长度
02 00 02 00     I帧控制域
```

控制域：

```text
VS = 1
VR = 1
```

ASDU 固定头：

```text
01        TypeId = 单点遥信
04        VSQ = 4 个信息体，非连续地址
14 00     COT = 响应总召唤
01 00     CommonAddress = 1号站
```

后面 4 个遥信点：

```text
03 00 00 00
05 00 00 00
08 00 00 01
09 00 00 00
```

对应：

| IOA | SIQ | 简化含义  |
| --: | --: | --------- |
|   3 |   0 | 3号点 = 0 |
|   5 |   0 | 5号点 = 0 |
|   8 |   1 | 8号点 = 1 |
|   9 |   0 | 9号点 = 0 |

整体含义：

```text
1号站响应总召唤，上送了4个单点遥信：
3号点=0，5号点=0，8号点=1，9号点=0。
```

### 13.3 双点遥信 TypeId = 03

双点遥信信息体结构：

```text
IOA 3字节 + DIQ 1字节
```

先只看低两位：

```text
DIQ & 0x03
```

常见含义：

|    值 | 含义            |
| ----: | --------------- |
| `0` | 不确定 / 中间态 |
| `1` | 分 / 开         |
| `2` | 合 / 关         |
| `3` | 不确定 / 故障态 |

双点遥信比单点遥信更适合表示开关位置，因为它能表达“不确定”。

## 14. 常用 TypeId：遥测

遥测就是数值量，例如：

```text
电压
电流
温度
功率
CPU 使用率
磁盘占用率
通信延迟
```

遥信和遥测的区别：

```text
遥信：状态，0/1/分/合/在线/离线
遥测：数值，12.5、220.3、98.6
```

常见遥测 TypeId：

| TypeId | 名称       | Value 部分                |
| -----: | ---------- | ------------------------- |
| `09` | 归一化遥测 | 2 字节值 + 1 字节品质     |
| `11` | 标度化遥测 | 2 字节整数 + 1 字节品质   |
| `13` | 短浮点遥测 | 4 字节 float + 1 字节品质 |

以 `TypeId = 09` 为例，信息体结构：

```text
IOA 3字节 + NVA 2字节 + QDS 1字节
```

含义：

| 字段    | 含义     |
| ------- | -------- |
| `IOA` | 点号     |
| `NVA` | 遥测值   |
| `QDS` | 品质描述 |

所以一个遥测点长度是：

```text
3 + 2 + 1 = 6 字节
```

---

## 15. 遥测帧示例

示例：

```text
68 13 06 00 02 00 09 82 14 00 01 00
01 07 00 A1 10 00 89 15 00
```

ASDU 固定头：

```text
09        TypeId = 归一化遥测
82        VSQ
14 00     COT = 响应总召唤
01 00     CommonAddress = 1号站
```

这里关键是：

```text
VSQ = 0x82
```

拆 VSQ：

```text
0x82 & 0x7F = 2
```

表示：

```text
有 2 个遥测点
```

最高位是 1，表示：

```text
连续地址
```

所以后面不是：

```text
IOA1 + Value1
IOA2 + Value2
```

而是：

```text
起始 IOA + Value1 + Value2
```

剩余内容：

```text
01 07 00 A1 10 00 89 15 00
```

拆成：

```text
01 07 00    起始 IOA = 0x701
A1 10 00    第 1 个遥测值 + 品质
89 15 00    第 2 个遥测值 + 品质
```

因为是连续地址，所以对应：

|       IOA | 原始值字节 | 品质   |
| --------: | ---------- | ------ |
| `0x701` | `A1 10`  | `00` |
| `0x702` | `89 15`  | `00` |

协议层能知道：

```text
1号站 0x701 遥测点 = A1 10
1号站 0x702 遥测点 = 89 15
```

至于它是“电压”还是“温度”，要查点表。

## 16. 点表查询方式

查点表时，通常不是只查 IOA，而是查：

```text
CommonAddress + IOA
```

也就是：

```text
哪个站 + 哪个点
```

例如：

```text
CommonAddress = 1
IOA = 0x701
```

点表里可能写：

| 公共地址 |   IOA | 类型 | 名称    | 单位 | 倍率 |
| -------- | ----: | ---- | ------- | ---- | ---: |
| 1        | 0x701 | 遥测 | A相电压 | kV   | 0.01 |

这样才能知道：

```text
0x701 这个遥测点 = A相电压
```

## 17. 品质 Quality

品质表示：

```text
这个值可不可信、是不是正常采集来的。
```

104 不只是传值，还会传这个值的状态。

例如遥测信息体：

```text
01 07 00 A1 10 00
```

拆成：

```text
01 07 00    IOA = 0x701
A1 10       遥测原始值
00          品质 QDS
```

这里品质是：

```text
00 = 正常
```

常见品质标志：

| 标志   | 含义                         |
| ------ | ---------------------------- |
| `IV` | Invalid，无效值              |
| `NT` | Not topical，非当前值 / 旧值 |
| `SB` | Substituted，人工替代值      |
| `BL` | Blocked，被闭锁              |
| `OV` | Overflow，溢出，主要用于遥测 |

工程上最重要的是：

```text
IV = 1 → 这个值无效，不应该当正常值用
NT = 1 → 这个值不是最新的
SB = 1 → 这个值可能是人工置入 / 替代的
BL = 1 → 这个点被闭锁
OV = 1 → 遥测值溢出
```

当前理解：

```text
IOA = 哪个点
Value = 点的值
Quality = 这个值能不能信
```

解析遥测时，不应该只保存数值，也应该保留品质：

```csharp
public sealed class MeasuredPoint
{
    public int IOA { get; init; }
    public double Value { get; init; }
    public byte Quality { get; init; }
}
```

## 18. 遥控暂时跳过

遥控属于下行控制命令，风险较高。

常见遥控包括：

```text
分闸
合闸
压板投入
压板退出
复归
启动
停止
```

遥控方向：

```text
主站 → 子站 / 装置
```

和遥信、遥测方向不同：

```text
遥信 / 遥测：子站上送给主站
遥控：主站下发给子站
```

本阶段先不深入遥控，只记住：

```text
S 帧只确认“我收到这帧”
遥控确认 ASDU 才表示“命令被接受 / 执行结果”
```

当前项目如果暂时不做遥控，可以优先跳过。

## 19. 总召唤流程

总召唤用于初始化获取全量数据。

整体流程：

```text
TCP 连接
  ↓
STARTDT act / con 启动传输
  ↓
主站发总召唤
  ↓
子站返回总召唤确认
  ↓
子站上送遥信 / 遥测 / 遥脉等全量数据
  ↓
子站返回总召唤结束
```

总召唤可以理解为：

```text
主站：把你当前所有遥信、遥测数据都发给我
子站：好的，我分批上送
```

总召唤不是请求一条数据，而是请求当前全量点数据快照。

---

## 20. 总召唤请求、确认、结束

总召唤相关报文的 `TypeId` 都是：

```text
64
```

也就是：

```text
TypeId = 0x64 = 总召唤
```

它们主要通过 `COT` 区分。

### 20.1 总召唤请求

主站发送：

```text
64 01 06 00 01 00 00 00 00 14
```

关键字段：

```text
64      TypeId = 总召唤
06 00   COT = 激活
14      QOI = 总召唤
```

含义：

```text
主站请求开始总召唤。
```

### 20.2 总召唤确认

子站返回：

```text
64 01 07 00 01 00 00 00 00 14
```

关键字段：

```text
07 00   COT = 激活确认
```

含义：

```text
子站确认：我收到总召唤请求，开始处理。
```

### 20.3 总召唤结束

子站最后返回：

```text
64 01 0A 00 01 00 00 00 00 14
```

关键字段：

```text
0A 00   COT = 激活终止
```

含义：

```text
子站通知：这次总召唤数据已经发完。
```

对照表：

| 报文类型   | TypeId |       COT | 含义     |
| ---------- | -----: | --------: | -------- |
| 总召唤请求 | `64` | `06 00` | 激活     |
| 总召唤确认 | `64` | `07 00` | 激活确认 |
| 总召唤结束 | `64` | `0A 00` | 激活终止 |

一句话：

```text
TypeId=64 说明这是总召唤；
COT=07 是确认；
COT=0A 是结束。
```

## 21. 变化数据主动上送

总召唤用于初始化，后续运行过程中一般不再反复全量召唤。

正常模式是：

```text
启动通信
  ↓
总召唤拿全量快照
  ↓
后续监听变化上送
```

变化上送可以理解成：

```text
变化上送 = 增量推送
```

也就是某个点发生变化后，子站主动上送变化点。

---

## 22. 变化上送看 COT

变化上送主要看 `COT`。

例如遥信变化上送：

```text
TypeId = 01 / 03
COT = 03 00
```

含义：

```text
TypeId = 01  单点遥信
TypeId = 03  双点遥信
COT = 03     突发 / 自发 / 变化上送
```

和总召唤返回数据对比：

| 场景                 | TypeId |       COT | 含义       |
| -------------------- | -----: | --------: | ---------- |
| 总召唤返回单点遥信   | `01` | `14 00` | 响应总召唤 |
| 变化主动上送单点遥信 | `01` | `03 00` | 突发变化   |
| 总召唤返回双点遥信   | `03` | `14 00` | 响应总召唤 |
| 变化主动上送双点遥信 | `03` | `03 00` | 突发变化   |

核心判断：

```text
TypeId 说明“是什么点”
COT 说明“为什么发”
```

## 23. 变化遥信示例

示例：

```text
68 0E 16 00 06 00 01 01 03 00 01 00 03 00 00 00
```

ASDU 拆解：

```text
01        TypeId = 单点遥信
01        VSQ = 1 个信息体
03 00     COT = 突发 / 变化上送
01 00     CommonAddress = 1号站
03 00 00  IOA = 3
00        SIQ = 0
```

含义：

```text
1号站的 3 号单点遥信发生变化，当前值为 0。
```

## 24. TESTFR 测试帧 / 心跳

104 是 TCP 长连接，但 TCP 连接“看起来还在”不代表对端真的还能正常响应。

所以空闲一段时间后，双方可以发测试帧。

测试帧可以理解为：

```text
我还活着，你还活着吗？
```

对方返回：

```text
我还活着。
```

测试帧属于 U 帧，没有 ASDU，是 6 字节短帧。

测试请求：

```text
68 04 43 00 00 00
```

测试确认：

```text
68 04 83 00 00 00
```

拆解：

```text
68              启动符
04              后面 4 字节
43 00 00 00     TESTFR act，测试请求
```

返回：

```text
68              启动符
04              后面 4 字节
83 00 00 00     TESTFR con，测试确认
```

## 25. TESTFR 和 S 帧的区别

| 报文    | 作用               |
| ------- | ------------------ |
| S 帧    | 确认收到对方 I 帧  |
| TESTFR  | 测试链路是否还活着 |
| STARTDT | 启动数据传输       |

区别：

```text
S 帧关心：数据有没有收到
TESTFR 关心：连接是否还可用
```

工程上通常要维护几个状态：

```text
LastReceiveTime
LastSendTime
WaitingTestConfirm
```

常见逻辑：

```text
多久没收到任何报文 → 发 TESTFR act
发了 TESTFR act 后多久没收到 TESTFR con → 认为链路异常
```

一句话：

```text
总召唤和变化上送负责数据；
TESTFR 测试帧负责长连接保活。
```

## 26. 当前已形成的 104 主流程

当前已经可以把 104 主流程理解成：

```text
TCP 连接
  ↓
STARTDT act / con 启动传输
  ↓
总召唤获取全量点数据
  ↓
解析遥信 / 遥测
  ↓
后续接收变化上送
  ↓
空闲时 TESTFR 保活
  ↓
必要时 S 帧确认 I 帧
```

其中：

```text
总召唤 = 初始化全量快照
变化上送 = 后续增量更新
TESTFR = 长连接保活
S 帧 = I 帧接收确认
```

## 27. 当前阶段常用判断表

| 看到什么                        | 先判断什么         |
| ------------------------------- | ------------------ |
| `68 04 07...`/`68 04 0B...` | U 帧启动           |
| `68 04 43...`/`68 04 83...` | U 帧测试           |
| `68 04 01...`                 | S 帧确认           |
| `TypeId=64`                   | 总召唤             |
| `COT=06/07/0A`                | 激活 / 确认 / 结束 |
| `COT=14`                      | 响应总召唤         |
| `COT=03`                      | 变化主动上送       |
| `TypeId=01/03`                | 遥信               |
| `TypeId=09/11/13`             | 遥测               |
| `CommonAddress + IOA`         | 查点表定位业务含义 |

## 28. 当前阶段核心结论

到目前为止，104 可以先按下面的模型理解：

```text
Frame 层：
  解决 I / S / U 帧识别、序号、确认、启动、保活

ASDU 层：
  解决 TypeId、VSQ、COT、CommonAddress、IOA、Value、Quality

点表层：
  解决 CommonAddress + IOA 对应什么业务含义
```

最核心的一句话：

```text
104 报文本身只传“哪个站、哪个点、什么值、这个值是否可信、为什么发”；
业务系统要靠点表把这些点解释成真实设备、状态或测量量。
```
